# Clock-Out — 2026-03-13 — 1001-VIDEO AVATAR

## 1) Objective Recap
The primary goal today was to ensure stability for the entire TalkFlow Video Avatar ML pipeline on the AWS EC2 g5.xlarge GPU deployment. We needed to ensure that MuseTalk, GFPGAN enhancement, and importantly, XTTS Voice Cloning natively integrate end-to-end without Out-Of-Memory errors or unintended fallbacks to static images / generic robotic voices.

## 2) Key Achievements & Fixes
### 🗣️ XTTS Voice Cloning (Audio-to-Video) Rescue
- **Fixed `torchcodec` Crash:** The latest ML stack dropped native decoding support for `.MP3` files without `torchcodec`. This caused Torchaudio (XTTS's backend) to fail silently and permanently skip user-selected voices, resulting in the app falling back to a generic robotic voice (`gTTS`).
  - Added `torchcodec` to `requirements-ml.txt`.
- **Pre-Processing Audio Conversions:** Rewrote the `tts.py` adapter to force FFmpeg decoding of `.MP3` to `.WAV` at 22,050Hz before XTTS processes it, bypassing any decoding anomalies entirely.
- **Fixed Celery CUDA Fork Issue:** Loading PyTorch/CUDA inside standard Celery subprocesses (`fork`) caused the system crash (`Cannot re-initialize CUDA in forked subprocess`). We updated the `docker-compose.yml` to lock the Worker into main-thread mode (`--concurrency=1 --pool=solo`), explicitly solving the crash and enabling GPU inferencing for cloning.

### ⚙️ Pipeline Optimization & Server Maintenance
- **Resolved Docker Disk Quotas:** EC2 server had run out of space ("no space left on device"). Performed a deep cache purge (Conda, PIP, Docker builder) removing >60 GB of stale volumes to allow rebuilds.
- **Dependencies Lockdown:** Pinned stable versions for `face-alignment` and `filterpy` inside `requirements-ml.txt` after upstream breaks, and wiped broken dependencies such as `mmcv`.
- **Nginx & Backend Integration:** Resolved an Nginx caching issue (502 Bad Gateway) after container restarts by forcing Nginx to re-resolve the backend's dynamic internal IP address.
- **Data Validation Insight:** Discovered that older Avatars lacked `voice_url` entries in PostgreSQL, which correctly forced the systemic skip logic. Educated the workflow on using newly minted Avatars to ensure the database correctly injects the target voices. 

## 3) Current Status
- **End-to-End Success:** The pipeline natively receives text along with an `.MP3` voice, successfully spawns XTTS inside the Celery Solo pool, generates the `.WAV`, feeds it to MuseTalk, lip-syncs the imagery, and packages the final composite successfully without falling back to a static image or a default generic voice.

## 4) Repository Info

- **Deployment:** Docker + Celery Worker (Pool=Solo) + AWS EC2
